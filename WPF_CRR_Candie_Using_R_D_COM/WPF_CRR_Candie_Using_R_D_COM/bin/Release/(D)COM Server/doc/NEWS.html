  • use package rscproxy instead of DLL rproxy.dll for connecting to R
  • use sciproxy.dll for connecting to Scilab 4.x or 5.x
  • RExcel distributed separately now
  • R/Scilab (D)COM Server is now distributed under a new license. See ../SC_PUBLIC for more information.

CHANGES in R(D)COM 2.50

  • Various smaller bugfixes.
  • Support for Scilab 4.1 in addition to R
  • RExcel: R started from the RExcel menu reads .RProfile from its own directory. This allows additional configuration options for R started from RExcel.
  • RExcel: GetArrayToVBA and PutArrayFromVBA allow to transfer data between VBA and R without putting them on a worksheet.
  • RExcel: setting R's working on R startup now respects if called from RExcel or called from outside RExcel.
  • RExcel configuration can be copied to tge clipboard from the dialog box.
  • RExcel: column names for data frames are coerced to valid variable names in R.
  • RExcel: some more configuration options added.
  • RExcel: Excel slider demo shows how to use RExcelStart.RS

CHANGES in R (D)COM 2.01

  • The installer now provides a new category excelsamples. This allows an installation of the (D)COM Server and RExcel with only the samples for RExcel.
  • Documentation enhancements: setup command line parameters and integration into 3rd party application setup.

CHANGES in R (D)COM 2.00

  • R (D)COM 2.00 requires R >= 2.2.0.
  • Data transfer now allows to transfer arrays of VT_BYREF types.
  • Data transfer supports arrays of VARIANTs of mixed types, e.g. now you can transfer an array with a boolean (VT_BOOL), an integer (VT_I4) and a string (VT_BSTR) at once. This works for both directions (SetSymbol() and GetSymbol()).
  • Special values (e.g. NA, NaN) are now correctly transferred using R (D)COM 2.00. The encoding used is borrowed from Excel.
  • Basic support for transfer of COM objects into R and from R are provided. VT_DISPATCH is used. On the R side, the package rcom (version >= 1.2) is required
  • Fully compatible COM server is also provided in the package rcom which can also be found on CRAN. rcom leaves the R GUI visible.
  • A (D)COM client library is found in the package rcom. This is e.g. useful to process "callbacks" to your own COM objects.
  • Rcommander can now be run from R (D)COM applications.
  • A bug in the graphics device has been fixed. Now colored plots work as expected.
  • Installation of RExcel has changed. Uninstall works fine now.
  • RExcel transfers complex numbers and date and time data types between R and Excel.
  • RExcel has direct access to RCommander

CHANGES in R (D)COM 1.36

  • Added RServerManagerAdmin
  • Fixed setup for sample sources (MFC client sample)
  • Prepared some start-up debugging traces
  • Works with the rcom library, allowing to have a command line window on the R process communicating with Excel. R server is a foreground process.
  • Support of R processes running under control of RServerManager. R server is attached to Excel from a server pool. Different instances of Excel (running on different machines) may access the same R process with the same data.
  • Improved stability and better error handling (error und debugging messages now go into separate modeless window working simultaneously with Excel)

CHANGES in 1.2

  • Loading of rproxy.dll has changed now. Previously loading failed if an outdated %R_HOME% setting was present, now the loading follows this algorithm:

    1. if %R_HOME% is set, try to load %R_HOME%\bin\rproxy.dll
    2. if the DLL has not been loaded, try to locate the current R installation
    3. by querying HKEY_LOCAL_MACHINE\SoftwareR-coreR\InstallPath, load from bin\rproxy.dll from the location this key points to
    4. if the DLL has not been loaded now, try to load rproxy.dll from %PATH%
    5. if this fails, return SCN_E_LIBRARYNOTFOUND.

      Every load failure is traced using OutputDebugString(). Use dbgview.exe to view this trace information. Get dbgview.exe from http://technet.microsoft.com/en-us/sysinternals/bb896647.aspx

  • Excel-Add In has been added to setup.
  • Sample for using R with python.
  • RExcel addin changes:
    • New worksheet functions RApplyC, RApplyA, RApplyAC. So far, RApply(function,args) called a function and args were the values for the arguments taken from sheet ranges. This does not work for named arguments. The worksheet function RApplyA(function,argstring,..) uses just one (Excel) argument which has to be a string giving all the arguments for the R function call.
    • RApply can ony return scalars or matrices. If the function called returns an object, it cannot be put in the worksheet directly. Therefore, RApplyC(function,component,args) assumes that the function call returns an object and returns the component given by the second argument from this object. RApplyC("cor.test","p.value", ....) will return the p-value of the correlation test.
    • RApplyAC combines the functionality of RApplyA and RApplyC.
    • RCall is used to call a procedure with no return values (an important use is calling graphics procedures from the worksheet). RCallA uses the string argument mechanism describe for RApplyA. RCall and RCallA are part of Excel's recalculation mechanism, therefore graphs will be updated automatically when data is changed.
    • MakeArgs turns a 2 column list or 2 row list of argument names and argument expressions into a string suitable for RapplyA, RApplyAC, or RCallA.
